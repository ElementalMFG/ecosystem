Real doc-drift sources (in priority order)

1. "Smart pager" framing across README and PRD §1 understates the device and directly conflicts with your view. PRD §8 also flatly excludes "smartphone replacement" — that ban is broader than you actually want.
2. PRD §5 says "no LLM on-device v1.0" but architecture keeps ss_ai as a first-class component. Pick one.
3. "Live video micro-frames" in the master plan reads like video calling; F-INT-03/04 fine print says WASM plugin, post-v1.0. Language mismatch.
4. Omega BOM is "TBD RISC-V + Linux SoM" — cellular/SatCom are promised on a board that isn't pinned. Either commit or mark v1.0 = Lite + Alpha only.
5. Lite's half-duplex PTT + no battery-SoC aren't up-front in the vision docs — buyers will assume Alpha-level features.
6. "Router/hotspot/WAP/extender/universal node" is all built (EPIC-17) but PRD §2.3 doesn't use those words, so it reads narrower than it is.

Recommended actions — minimal set, no capability loss

Necessary (do these):
- RFC-SCOPE-LOCK (one T3 doc): reconcile pager naming, video, browsing, LLM, camera, Omega BOM. Freezes v1.0 truth so every downstream story stops drifting.
- PRD §1 + README rewrite (T4 batch): retire "pager" as primary noun; primary = "sovereign multi-band mesh communicator + universal node." Keep "pager form-factor" as descriptor.
- PRD §2.3 device-roles paragraph (T4): explicitly enumerate router / hotspot / WAP / bridge / gateway / extender / mesh-node, each linked to its F-HGW-/F-EUD- story. No new work — just naming.
- Resolve LLM contradiction (T4): either delete ss_ai from architecture until v2.x, or add a v1.0 F-AI-* story with a pinned model + perf gate. My call: keep ss_ai as capability-only scaffold, no product-level LLM promise in v1.0.
- Kill "live video micro-frames" ambiguity (T4): rename to "thumbnail attachments" OR promote to a real F-VID-* story with codec + bearer plan. Don't leave both.
- Pin Omega scope (T3): either commit BOM (SoM part + modem PN) or add "v1.0 ships Lite + Alpha; Omega is v1.x" callout in README.
- Add Lite constraints callout (T4): half-duplex PTT and no fuel-gauge visible in README variant matrix.

Optional simplifications that don't lose capability:
- Descope TypeScript + Dart SDKs to v1.1 (keep C + Rust + Python for v1.0). Saves 8 stories, no capability lost — apps use their native SDK.
- EPIC-13 Meshtastic clean-room compat: check if you need full v2.x parity or just tx/rx interop on a stated channel-set. Could halve the epic.
- ss_ai = STT/TTS only for v1.0, defer LLM to v2.x formally.

Things you specifically named that I'd add (small, real gaps):
- Wake-word (1 story under EPIC-14 — trivial with existing Whisper pipeline, high UX value for hands-free).
- Web browsing: I'd say no — an on-device browser on ESP32-P4 is a lot of surface for something the paired phone does better. The companion + IP-tether already covers "browse the web through the device."
- Video calling: I'd keep out of core and formalize as a signed WASM plugin post-v1.0. Real-time video over 1W mesh isn't the shape of this device; over HaLow/tether it's a plugin, not core.

Tradeoff to be aware of

If you widen v1.0 scope (add LLM, browser, video), the T1 crypto/secure-boot + certification epics (24 alone has ~35 stories) get harder to close on schedule. The RFC-SCOPE-LOCK is the lever — decide once, in writing, and every epic falls in line.
