- **Only `ss_power_core.c` is host-built** (`firmware/test/host/CMakeLists.txt:53`); the IDF glue `ss_power.c` (esp_sleep_* calls) is NOT. So the record-vs-platform divergence logic (`ss_power_wake_timer_set/clear`, init disable, staging/error propagation) has ZERO automated coverage — review it by reasoning against IDF, don't trust the host green. AC for these stories only requires the pure core host-tested, so a coverage gap here is not a ding by itself.
- **IDF `esp_sleep_disable_wakeup_source()` return semantics (v5.3.5, VERIFIED against sleep_modes.c):** `CHECK_SOURCE(src,val,mask)` is `((s_config.wakeup_triggers & mask) && (source==value))` — it requires the trigger bit to be currently SET. So `esp_sleep_disable_wakeup_source(ESP_SLEEP_WAKEUP_TIMER)` on a NOT-armed timer fails every CHECK_SOURCE branch, falls to the final else, `ESP_LOGE`s "Incorrect wakeup source (%d) to disable.", and returns **ESP_ERR_INVALID_STATE** (not ESP_ERR_INVALID_ARG, not ESP_OK). Consequences: (1) calling it unconditionally for idempotency logs an error line on every not-armed clear — gate on your own record instead; (2) absorbing ESP_ERR_INVALID_STATE is load-bearing, NOT dead code. CORRECTION: an earlier version of this memory wrongly claimed it always returns ESP_OK — I asserted that from memory without fetching; the coordinator's counter-evidence was right. See [[reference-idf-source-fetch]].
- **`esp_sleep_enable_timer_wakeup(us)` (v5.3.5, VERIFIED) validates-before-mutate:** if `us` exceeds the LP-timer max it returns ESP_ERR_INVALID_ARG BEFORE setting `wakeup_triggers`/`sleep_duration` — so there is no partial-arm, and a "commit the record only after a clean platform accept" pattern keeps record⟺platform in lockstep. S3 platform max ≈ 2.1e15 us; the 30-day `SS_PWR_WAKE_TIMER_MAX_US` cap (~2.6e12 us) is ~800x below it, so no contract-valid value is platform-rejected.
- **Timer-wake stickiness is correctly reasoned:** `esp_sleep_enable_timer_wakeup(us)` stores `sleep_duration` in IDF's `s_config`; the RTC alarm is computed at each `esp_light_sleep_start()`/`esp_deep_sleep_start()`, and triggers persist across light-sleep cycles until disabled. So set-time arming legitimately yields the contract's "countdown from each sleep entry + sticky-until-clear." Deep-sleep/hibernate wake is a reset → `ss_power_init` re-zeros everything.
- **HIBERNATE == DEEP_SLEEP in the glue:** both map to `esp_deep_sleep_start()` with no `esp_sleep_pd_config()` power-down. Timer wake works in both, so the frozen header's "timer applies to HIBERNATE" holds for the current impl. If a future change makes HIBERNATE a true power-down, EXT0/EXT1 GPIO wake would stop while timer keeps working — the "whichever of timer/GPIO fires first" bullet would then mislead for HIBERNATE.
